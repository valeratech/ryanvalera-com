# ADR-005 — Cloudflare R2 as Future Media and Artifact Storage Layer

## Status

Deferred — post-Milestone 6D

## Date

June 2026

---

## Context

The current platform architecture serves the portfolio site exclusively through GitHub Pages (primary) and Cloudflare Pages (secondary), with Cloudflare as the front door. This architecture handles HTML, CSS, JS, and image assets well.

As the portfolio matures, two categories of content will need a dedicated delivery mechanism that GitHub Pages is not well suited for:

1. **Demonstration media** — short video clips showing Orthanc PACS configuration, DICOM C-STORE operations, Wireshark protocol analysis, and DCMTK command-line workflows
2. **Technical artifacts** — architecture diagrams, project PDFs, DICOM workflow documentation, and downloadable case studies

GitHub Pages has a 1GB soft limit on repository size and is not designed for binary media delivery. Storing large media assets directly in the git repository creates performance and maintainability problems over time.

---

## Decision

Introduce Cloudflare R2 as a dedicated object storage layer for portfolio media and technical artifacts, served through a separate subdomain (`media.ryanvalera.com`).

R2 will not replace GitHub Pages as the primary website origin or Cloudflare Pages as the secondary failover origin. It is an additive component that handles content the primary architecture is not designed for.

---

## Rationale

### Why R2 Specifically

- Native Cloudflare integration — no egress fees for bandwidth delivered through Cloudflare's network
- Custom domain support via `media.ryanvalera.com`
- Consistent with the project's goal of demonstrating the breadth of Cloudflare's platform
- S3-compatible API — familiar tooling for uploads and automation
- Pairs naturally with the GitHub Actions CI/CD pipeline already planned for Milestone 6E

### Why a Separate Subdomain

Separating `media.ryanvalera.com` from `ryanvalera.com` cleanly isolates:

- Cache policies (media assets have different TTL requirements than HTML)
- Access controls (R2 bucket policies separate from site delivery)
- Cost tracking (R2 usage visible independently from Pages/CDN usage)
- Future automation (media upload workflows don't interfere with site deployment workflows)

### Why This Is Deferred

R2 integration requires:

- Cloudflare Load Balancing operational (Milestone 6D)
- GitHub Actions CI/CD pipeline established (Milestone 6E)
- Actual demonstration content recorded and anonymized

None of these prerequisites are currently complete. Implementing R2 before the core infrastructure is stable would add complexity without delivering value.

---

## Architecture Impact

This phase expands the platform's Cloudflare component coverage to include:

```text
Current:
  DNS, Proxy/CDN, SSL/TLS, Cache Rules, Transform Rules,
  Security Headers, WAF, Bot Protection, Rate Limiting,
  Analytics, Pages, Load Balancing, Health Checks

Added by R2 phase:
  R2 Object Storage
  Custom subdomain routing
  Media delivery cache policy
  Artifact governance
  GitHub Actions media upload automation
```

---

## Consequences

- Repository size remains manageable — binary media assets are not stored in git
- Demonstration content becomes linkable from `profile.html` project panels
- Introduces a new Cloudflare billing component (R2 storage + operations)
- Requires anonymization protocol for any DICOM/PACS demonstration material
- Adds `media.ryanvalera.com` DNS record and R2 bucket to operational inventory
