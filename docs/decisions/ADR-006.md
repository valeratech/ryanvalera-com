# ADR-006 — Canonical Origin Enforcement vs. Direct-Reachability Fallback

## Status

Accepted

Supersedes the direct-reachability / fallback claims in ADR-001 and ADR-003 (those records to be corrected accordingly). Live testing (July 2026, recorded below) established that the primary origin already enforces canonical redirection natively; the secondary origin's browser-bypass seam was closed via an account-level Bulk Redirect, verified in production.

## Date

July 2026

---

## Context

Traffic to `ryanvalera.com` resolves through Cloudflare DNS and is routed by the Cloudflare Load Balancer to one of two origins:

```text
ryanvalera.com  →  Cloudflare DNS  →  Cloudflare Load Balancer
                                          ├─ github-pages-primary   (Host: ryanvalera.com)      → valeratech.github.io
                                          └─ cloudflare-pages-secondary (Host: pages.ryanvalera.com) → pages.ryanvalera.com → ryanvalera-com.pages.dev
```

The two pools deliberately send **different Host headers** — `ryanvalera.com` to the primary, `pages.ryanvalera.com` to the secondary. This asymmetry was established in failover Test 2/3 (June 2026, see `docs/runbooks/load-balancer.md`): Cloudflare Pages rejects a foreign Host header, so the secondary pool must address the origin by the hostname it is actually configured to serve.

The concern that prompted this ADR: each origin is also reachable **directly**, bypassing the Load Balancer, and a direct request is served by that platform's own defaults with **none** of the Cloudflare edge controls (WAF, Bot Fight Mode, Transform Rules, cache governance, deferred HSTS/CSP) applied. For a portfolio whose thesis is platform engineering, an uncontrolled copy of the site reachable off-canonical is a presentation and architecture-integrity seam — low severity (static payload, no backend/auth/API), but a visible one to the intended audience.

---

## Live Findings (July 2026)

Direct measurement, not inference, established the actual state of each path:

| Path | Request | Result | Interpretation |
|---|---|---|---|
| Primary, browser bypass | `GET valeratech.github.io/ryanvalera-com/` | **301 → `https://ryanvalera.com/`** | Already enforced to canonical |
| Primary, LB failover | `GET valeratech.github.io/` · Host `ryanvalera.com` | **200**, body validates | Failover path healthy; redirect does not affect it |
| Secondary, LB failover | `GET pages.ryanvalera.com/` · Host `pages.ryanvalera.com` | **200**, body validates | Failover path healthy |
| Secondary, foreign Host | `GET ryanvalera-com.pages.dev/` · Host `ryanvalera.com` | **403** | Cloudflare Pages rejects foreign Host — partial self-protection |
| **Secondary, browser bypass** | `GET ryanvalera-com.pages.dev/` (native Host) | **200, serves the site** | **The one genuine open seam** |

Two findings materially changed this ADR from its initial draft:

**1. The primary origin already enforces canonical, and does so failover-safe.** The GitHub Pages custom-domain redirect is **Host-aware**, not unconditional: a browser hitting the naked URL is 301'd to canonical, while the Load Balancer's health-check and failover traffic (Host: `ryanvalera.com`) receives a clean 200. The draft of this ADR asserted GitHub's custom-domain feature issues an *unconditional* 301 that would strand failover — **that assertion was empirically false for this configuration** and has been removed. The mechanism previously flagged as dangerous is in fact the one already protecting the primary origin.

**2. The fallback described in ADR-001/003 does not function as documented.** Those ADRs describe `valeratech.github.io/ryanvalera-com/` as a genuine Cloudflare-independent fallback URL during a Cloudflare outage. In practice that URL now 301-redirects to `ryanvalera.com`, which resolves through Cloudflare — so it is not a working escape hatch. The documented fallback is already, in effect, superseded by the redirect; ADR-001/003 describe a resilience property that no longer holds. This should be corrected in those records when this ADR is accepted.

---

## The Remaining Question

Only the secondary origin's browser-bypass path is open: `ryanvalera-com.pages.dev` serves the site directly, with no edge controls. It already rejects a foreign `Host: ryanvalera.com` (403), so it is partially self-protecting; what it serves is browsers using its own native `.pages.dev` Host.

Enforcing canonical here is **provably failover-safe by construction.** The Load Balancer reaches the secondary via the *proxied* hostname `pages.ryanvalera.com`; a redirect scoped to the *raw* hostname `ryanvalera-com.pages.dev` therefore cannot intersect failover traffic — the two use different Host headers by the deliberate design established in Test 2/3. This is the same asymmetry that once caused a failover failure now guaranteeing clean enforcement.

The fallback-vs-seam tension that dominated this ADR's initial framing has largely dissolved: the primary already enforces canonical without cost to failover, and the secondary's fix carries no failover risk. What remains is a single low-consequence action, not a genuine dilemma.

---

## Decision

**Enforce canonical origin on the secondary via a Host-scoped redirect. No change to the primary (already enforced) or to Load Balancer configuration.**

Mechanism — as implemented and verified (July 2026):

A zone-level Redirect Rule scoped to `http.host eq "ryanvalera-com.pages.dev"` was attempted first and **did not fire**. Cloudflare Pages serves its default `.pages.dev` hostname upstream of the zone's rule engine, so zone-level Redirect Rules never see that traffic. This is a genuine finding: **the default Pages hostname cannot be redirected by a zone Redirect Rule; it requires an account-level Bulk Redirect**, which intercepts before Pages serves.

The implemented fix:

```text
Cloudflare → Bulk Redirects
  List:   pages_dev_canonical
  Source: ryanvalera-com.pages.dev/
  Target: https://ryanvalera.com/   (301)
  Params: subpath matching ON, preserve path suffix ON,
          preserve query string ON, include subdomains OFF
  Rule:   pages_dev_canonical_rule  (activates the list)
```

Constraints observed:

- The Bulk Redirect matches the **raw** hostname `ryanvalera-com.pages.dev` only. It does **not** touch `pages.ryanvalera.com` — the Load Balancer's failover Host — because that is a distinct custom domain, not a subdomain of the matched host, and `include subdomains` is off.
- `include subdomains` is off deliberately, so preview deployment URLs (`<hash>.ryanvalera-com.pages.dev`) are not swept in.
- The primary origin required no action; its Host-aware 301 already satisfied this decision.

Post-change verification (production):

| Path | Result | Meaning |
|---|---|---|
| `ryanvalera-com.pages.dev/` | `301 → https://ryanvalera.com/` | Seam closed |
| `ryanvalera-com.pages.dev/media.html?project=sentinel` | `301 → …/media.html?project=sentinel` | Path + query preserved |
| `pages.ryanvalera.com/` (Host: pages.ryanvalera.com) | `200`, body validates | Failover path unaffected |

Both origins now present canonical behavior to browsers; neither Load Balancer failover path is affected.

---

## Consequences

- After enforcement, both origins present canonical behavior to browsers: primary 301s already; secondary 301s via the new rule. Neither Load Balancer failover path is affected.
- The Cloudflare-independent fallback described in ADR-001/003 is acknowledged as non-functional (already redirected); those ADRs are corrected rather than left to imply a resilience property that does not exist. A genuinely provider-independent fallback, if later desired, is a separate design decision (and was explicitly out of scope per ADR-001's Cloudflare-focus rationale).
- The deferred security headers (HSTS/CSP) harden the canonical edge; once both origins redirect to canonical, there is no uncontrolled direct-serve path left for those headers to miss. This ADR and the header work remain distinct but now compose cleanly.
- Posture register: a single entry referencing this ADR, classified as an **accepted, revisited trade-off** once enforcement lands, `revisit: false` thereafter. Until then, `kind: accepted-tradeoff`, `revisit: true`.
