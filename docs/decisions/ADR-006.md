# ADR-006 — Canonical Origin Enforcement vs. Direct-Reachability Fallback

## Status

Proposed

Revisits the direct-reachability posture accepted in ADR-001 and ADR-003. Those ADRs remain in force until this decision is accepted; this record does not silently overturn them.

## Date

July 2026

---

## Context

Traffic to `ryanvalera.com` resolves through Cloudflare DNS and is routed by the Cloudflare Load Balancer to one of two origins:

```text
ryanvalera.com  →  Cloudflare DNS  →  Cloudflare Load Balancer
                                          ├─ github-pages-primary  → valeratech.github.io
                                          └─ cloudflare-pages       → ryanvalera-com.pages.dev
```

Both origins are also reachable **directly**, bypassing the Load Balancer entirely:

- `https://valeratech.github.io/ryanvalera-com/`
- `https://ryanvalera-com.pages.dev`

ADR-001 and ADR-003 documented this direct reachability as an **accepted trade-off** and, more specifically, as a **deliberate resilience feature**: the GitHub Pages URL is described as "a genuine fallback URL during Cloudflare platform events." The reasoning was sound for its purpose — during a total Cloudflare outage (DNS + LB + Pages origin all affected simultaneously, per ADR-001), the naked GitHub Pages URL is the one path to the site that does not depend on Cloudflare.

Two things have since made this worth revisiting.

**1. The direct origins do not carry the edge control plane.** Every control this project implements — WAF, Bot Fight Mode, Response Header Transform Rules, cache governance, and the deferred HSTS/CSP headers — is evaluated at the Cloudflare edge, keyed to `ryanvalera.com`. A request to `valeratech.github.io` or `ryanvalera-com.pages.dev` is served by that platform's own defaults with none of these controls applied. The direct origins are therefore a copy of the site with the entire security and governance layer stripped away.

**2. For a portfolio whose thesis is platform engineering, that stripped copy is a visible seam.** The audience this site is built for can open a naked origin, inspect response headers, and observe that the security posture the project documents is simply absent there. On a portfolio demonstrating *how* the author engineers, an unproxied origin serving the same content without the edge controls undercuts the demonstration more than it reinforces it.

The severity is low in traditional terms — the payload is static, with no backend, authentication, API, or sensitive resource to reach. This is not a confidentiality or integrity exposure. It is an **architecture-integrity and presentation** concern: the canonical, controlled path is not the enforced path.

---

## The Tension

The same property is simultaneously an asset and a liability, and the two cannot both be fully preserved on the GitHub Pages origin:

| Direct reachability as… | Value | Cost |
|---|---|---|
| **Resilience fallback** (ADR-001/003 stance) | Provider-independent escape hatch during a total Cloudflare outage | Serves the site without any edge controls; visible seam |
| **Security/architecture seam** (this ADR's concern) | — | Canonical controlled path is bypassable; demonstration weakened |

Enforcing canonical origin means redirecting the naked URLs to `ryanvalera.com`. But `ryanvalera.com` resolves **through Cloudflare**. A redirect that sends `valeratech.github.io` → `ryanvalera.com` therefore destroys precisely the out-of-Cloudflare fallback that ADR-001/003 valued: during a Cloudflare outage, the redirect target is itself unreachable.

**You cannot both enforce canonical origin and retain a Cloudflare-independent fallback on the same origin.** This is the real decision.

---

## Technical Finding — Failover Is Protectable

A critical implementation detail determines whether enforcement is even safe: **can the Load Balancer still fail over to an origin that redirects browsers to canonical?**

The Load Balancer pool `github-pages-primary` is configured with an explicit **`Host Header: ryanvalera.com`** (see `docs/runbooks/load-balancer.md`). Consequently, LB-originated traffic — both health checks and live failover requests — reaches GitHub Pages as:

```text
GET /
Host: ryanvalera.com          ← injected by the Load Balancer
(TCP connection to valeratech.github.io)
```

A browser bypassing the Load Balancer reaches the same origin as:

```text
GET /
Host: valeratech.github.io    ← the bypass path to be closed
```

**These two request classes are distinguishable by Host header.** A redirect scoped to fire only when `Host == valeratech.github.io` catches browser bypass while leaving LB failover traffic (`Host: ryanvalera.com`) untouched. The fire exit stays open; the front door is enforced.

**Critical caveat:** GitHub Pages' built-in *custom-domain* feature issues an **unconditional** 301 from the `github.io` URL that does **not** respect Host-header scoping. Using that feature would redirect LB failover traffic as well, stranding the failover path. The custom-domain setting is therefore the wrong mechanism. Enforcement must be done with a **Host-scoped rule** (Cloudflare-side, or an equivalent that inspects the Host header), never GitHub's custom-domain redirect.

This finding removes the availability risk from *one* side of the tension — but only for the mechanism that can distinguish Host headers. It does not resolve the fallback-vs-seam decision itself.

---

## Decision

**Deferred pending resolution of the fallback-vs-seam question, and pending a live failover test.**

This ADR records the tension and the technical finding; it does not yet commit to enforcement. Two things must happen before enforcement is applied:

1. **Resolve the stance.** Decide whether the GitHub Pages direct URL is retained as a resilience fallback (accept the seam, status quo) or enforced to canonical (close the seam, forfeit the Cloudflare-independent fallback). This is a judgment call weighing a low-severity presentation seam against a genuine escape hatch, for a project whose explicit goal (ADR-001) is demonstrating Cloudflare engineering rather than cross-provider redundancy.

2. **Test failover against a scoped redirect.** Before any redirect is applied in production, confirm empirically that a `Host: valeratech.github.io`-scoped 301 leaves Load Balancer health checks (which send `Host: ryanvalera.com`) passing and failover functional. Do not assume; verify.

The `ryanvalera-com.pages.dev` origin is a separate, lower-consequence case: it is the LB's *secondary* origin and was never load-bearing as a fallback. A Cloudflare Redirect Rule sending `*.pages.dev` → `ryanvalera.com` can be applied independently once the GitHub Pages decision is made, and carries no failover risk of its own.

---

## Consequences

- Until this decision is resolved, both direct origins remain reachable and uncontrolled; the seam documented here persists as a **known, accepted** condition rather than an unexamined gap.
- If enforcement is chosen, the GitHub Pages fallback described in ADR-001 and ADR-003 is forfeited; those ADRs must be updated to reflect the superseded stance, and the total-Cloudflare-outage scenario in ADR-001 loses its documented mitigation.
- If the fallback is retained, the security-header work (HSTS/CSP, currently deferred) hardens only the canonical edge; the naked origins remain a documented exception to that hardening, which should be stated explicitly wherever those headers are claimed.
- This decision is distinct from — and must not be merged with — the deferred security-header enhancements. Those harden the canonical edge; this decides whether the uncontrolled origins exist at all.
- The posture register should carry a single entry referencing this ADR, classified as an **accepted, revisited trade-off** (not an unmitigated gap), with `revisit: true`.
