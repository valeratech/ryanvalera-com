# ADR-001 — Cloudflare as Shared Control Plane

## Status

Accepted

## Date

June 2026

---

## Context

This platform uses Cloudflare as the front door for DNS resolution, SSL/TLS termination, CDN edge caching, WAF enforcement, bot protection, rate limiting, and Load Balancer routing between two origins (GitHub Pages and Cloudflare Pages).

Both the Load Balancer and one of the two origins (Cloudflare Pages) are Cloudflare infrastructure. This means Cloudflare serves as both the routing layer and one of the endpoints being routed to.

---

## Decision

Accept Cloudflare as the shared control plane for this platform, with the understanding that a major Cloudflare platform event could affect DNS resolution, Load Balancer routing, WAF enforcement, and access to the Cloudflare Pages origin simultaneously.

---

## Rationale

The goal of this project is to demonstrate Cloudflare platform engineering — DNS administration, SSL/TLS configuration, WAF, CDN, and Load Balancing — not to build cross-provider redundancy.

GitHub Pages remains independently reachable at `https://valeratech.github.io/ryanvalera-com/` if Cloudflare DNS is bypassed directly. It is not dependent on Cloudflare infrastructure to serve content.

Building true cross-provider redundancy (e.g., AWS CloudFront + S3 as a secondary origin) was evaluated and rejected due to SSL/TLS incompatibilities at the origin layer. See ADR-002 for details.

---

## Consequences

- A full Cloudflare platform outage would affect DNS, routing, and the secondary origin simultaneously
- GitHub Pages remains accessible directly during a Cloudflare outage
- This trade-off is appropriate given the project's goal of demonstrating Cloudflare administration
